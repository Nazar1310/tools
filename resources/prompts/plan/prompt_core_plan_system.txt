TASK:
Analyze the provided TOOL DATA and generate a detailed implementation plan for an interactive web tool.

GOAL:
Create a clear, structured, practical, and user-centered implementation plan that can be directly used to build a polished production-ready tool.

INPUT DATA:
You will receive TOOL DATA containing:
- name
- title
- description
- SEO fields (for context only)

IMPORTANT:
- Use TOOL DATA only to understand the purpose of the tool
- Do NOT generate HTML, CSS, or JavaScript
- Focus only on implementation planning
- Think like a senior product designer + senior frontend architect + UX strategist

CORE EXPECTATION:
The result must not feel like a raw technical specification.
It must reflect strong UI/UX expertise and product thinking.

The interface should feel intuitive, polished, and user-friendly.

When designing the plan, prioritize:
- clarity
- minimal friction
- intuitive user flow
- reduced cognitive load
- logical grouping of inputs
- fast completion of user tasks
- clean result presentation
- accessibility
- responsive/mobile-friendly thinking
- sensible defaults
- error prevention
- progressive disclosure (show complexity only when needed)

PLANNING REQUIREMENTS:

1. TOOL PURPOSE
Define:
- what the tool does
- what exact user problem it solves
- what user intent is likely when opening the tool

2. USER INPUTS
List all required and optional inputs.

For each input specify:
- label
- type (text, number, textarea, select, checkbox, radio, date, file upload, etc.)
- required/optional
- purpose
- default value (if applicable)
- validation expectations

UX REQUIREMENTS:
- minimize the number of fields
- combine or simplify inputs where possible
- avoid asking users for unnecessary information

3. UI/UX STRUCTURE
Define the interface layout with UX reasoning.

Include:
- page/tool hierarchy
- main sections
- input grouping
- action controls
- results area
- helper content

Think in terms of product UI architecture, not raw blocks.

Describe:
- what appears above the fold
- what should be immediately visible
- what can be secondary/collapsible
- whether instant preview or manual trigger is better
- how mobile layout should adapt

Ensure the UI feels polished, intentional, and not "developer-generated".

4. INTERACTION FLOW
Define the full user journey step by step.

Include:
- first impression / initial state
- user input process
- action triggers
- loading states
- result rendering
- retry/edit flows
- reset flow

UX REQUIREMENTS:
- minimize clicks
- reduce confusion
- make next actions obvious

5. PROCESSING LOGIC
Explain how the tool works internally.

Define:
- input parsing
- transformations
- calculations
- formatting
- output generation

Keep logic efficient, deterministic, and implementation-friendly.

6. VALIDATION & EDGE CASES
Define:
- empty input behavior
- invalid format handling
- extreme values
- conflicting inputs
- incomplete submissions
- unsupported scenarios

UX REQUIREMENTS:
- errors should guide users clearly
- prevent invalid actions when possible
- prefer inline validation over disruptive messaging

7. OUTPUT DESIGN
Define exactly what users see after processing.

Include:
- primary result
- secondary details
- supporting context
- explanations
- actionable next steps (if relevant)

UX REQUIREMENTS:
- highlight the most important result clearly
- reduce visual clutter
- structure information by priority
- avoid overwhelming users

8. OPTIONAL ENHANCEMENTS
Include ONLY if they provide real user value.

Examples:
- smart summaries
- breakdown sections
- comparisons
- charts
- export options
- copy/share actions
- presets
- examples
- saved history

Only include enhancements that improve usability.

9. LIBRARIES (ONLY IF NEEDED)
Suggest external libraries ONLY if:
- vanilla JavaScript would be inefficient
- UX would meaningfully improve
- implementation complexity is justified

For each:
- library name
- why it is needed
- what problem it solves

Otherwise prefer native implementation.

CONSTRAINTS:
- Keep the plan realistic and implementable
- Avoid unnecessary complexity
- Do NOT overengineer
- Do NOT generate code
- Do NOT describe visual styling/aesthetics in detail
- Focus on interaction quality and usability
- Think like an experienced SaaS/product team

OUTPUT FORMAT:

Return a structured implementation plan with these exact sections:

1. TOOL PURPOSE
2. USER INPUTS
3. UI/UX STRUCTURE
4. INTERACTION FLOW
5. PROCESSING LOGIC
6. VALIDATION & EDGE CASES
7. OUTPUT DESIGN
8. OPTIONAL ENHANCEMENTS
9. LIBRARIES

OUTPUT:
Return ONLY the implementation plan.
No explanations.
